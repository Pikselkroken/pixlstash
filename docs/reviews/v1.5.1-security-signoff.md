# CSO security sign-off — v1.5.1 (read-share-token BOLA)

**Decision: GO, conditional.** The emergency patch is approved to ship as **v1.5.1**
provided the release conditions below are met. This memo is the sign-off of record;
the full technical review is in [`bulk-token-scoping.md`](./bulk-token-scoping.md).

Source branch: `bulk-token-scoping`. Released baseline: `v1.5.0` (tag `e2cadf14`).

---

## What this patch fixes (HIGH — closed, tested, committed)

Broken object-level authorization (OWASP API1) reachable by a normally-issued
resource-scoped READ **share token**. A share-link holder could read data outside
the resource they were granted.

- Bulk POST reads (thumbnails, tag bulk-fetch, membership, guest-scores) — `b337aa20`
- `/pictures` listing/stream/count, tag GETs, project attachments/export/summary, character/set summary + name-lookups — `053d9bdb`
- Three sibling leaks found in review: `/pictures?character_id=UNASSIGNED` bypass, `/pictures/{id}/{field}`, stack read endpoints — `a97b871f`

**Verification:** `python -m pytest --fast-captions --force-cpu tests/test_read_token_security.py`
→ 51 passed, 2 failed. The 2 failures are `TestRateLimiter` timing tests, unrelated
to scoping; they fail because the rate-limiter fix lives on a different branch
(`rate-limiter-fix`), not because of this patch. Confirm the same 2-and-only-2 on
the release branch (see watch-outs).

Independent review: three reviewers (senior backend, QA, CSO-adversarial), each
finding reproduced live. They returned the three blockers above, which are now fixed.

---

## Accepted risk (CTO decision; CSO concurred)

Two read endpoints remain unscoped. Not vault content, credentials, or RCE — only
generation/score metadata, and only reachable by a holder of an already-issued
scoped READ token. Deferred to keep the emergency patch scope frozen.

| ID | Endpoint | Severity | Leak |
|----|----------|----------|------|
| R1 | `GET /comfyui/pictures/{id}/workflow` | MEDIUM | ComfyUI prompts/seeds/model names (metadata only) |
| R2 | `GET /pictures/{id}/character_likeness` | LOW | existence + likeness score |

**Owner:** CTO. **Revisit trigger:** fix both (same one-line `enforce_picture_scope`)
*before* scoped share links become a promoted or widely-used feature, in particular
before any are issued against the public demo. Take the fix opportunistically in any
normal release before then.

## WONTFIX (working as intended)

Stacking unions project/set membership across stack members (documented invariant in
`services/stack_membership.py`). A set-scoped share therefore exposes pictures stacked
into the set. Not exploitable — stacking is a write, READ tokens can't stack, so only
the owner can cause it, on their own data. Accepted as intended.

## Release conditions (must hold for the GO to stand)

1. **Cherry-pick, do not merge.** Cut `release-v1.5` from the `v1.5.0` tag and
   cherry-pick **only** the security payload. Do NOT merge `bulk-token-scoping`
   wholesale — it carries ~20 commits of unreleased feature work (Playwright, T2I,
   version-JSON, etc.) that must not ride into an emergency patch.

   Security payload, in order:
   - `b337aa20` Add scoping for bulk read tokens
   - `053d9bdb` Scope read-share tokens on the remaining GET reads
   - `a97b871f` Close three more read-token scope leaks

   (The doc commits — process notes and these review files — are records; they belong
   on `develop`/`main`, not on the release branch.)

2. **Changelog:** add a `[1.5.1] [Security:High]` entry.

3. **Re-run the security suite on the release branch** after cherry-picking and
   confirm the read-token isolation tests pass (see watch-outs for the rate-limiter
   tests).

4. **Bump version** to `1.5.1` and tag.

## Watch-outs for whoever cuts the release

- **Cherry-pick conflicts are possible.** The three security commits sit on top of
  intervening feature commits (`c3afe804` etc.) that are not in `v1.5.0`. If a
  security commit touches a file those features also changed, expect a conflict and
  resolve it carefully (keep the scope guards). If conflicts are heavy, an
  alternative is to reapply the guard changes by hand against the `v1.5.0` tree.
- **Rate-limiter tests.** `test_read_token_security.py` contains `TestRateLimiter`
  tests that depend on the rate-limiter fix (on `rate-limiter-fix`, not in `v1.5.0`).
  On the release branch they will likely fail for the same pre-existing reason.
  Decide deliberately: either cherry-pick the rate-limiter fix too, or accept those
  two as known-unrelated and gate the release on the *scope* tests
  (`TestResourceScopedReadTokenIsolation` and `TestReadTokenBlocks*`) passing.
- **Deferred test coverage.** The patch's tests cover the fixed holes, but the full
  review lists test gaps not yet filled (real cross-project attachment *download*,
  `/pictures/stream`, `export_project`, name-lookups). The code fixes are in place
  and were confirmed by reading the handlers; the missing tests are tracked
  follow-ups, not release blockers.
- **Separate hardening (not this patch):** the ALL+resource_type token footgun — see
  the full review. Not reachable by the share-token threat principal; track and fix
  in a normal release.
